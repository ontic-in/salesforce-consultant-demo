## Boundaries & Notes

### Common Gray Areas

| Request | Decision | Notes |
|---------|----------|-------|
| "We need Field Service Lightning" | Out of scope | Separate FSL implementation project |
| "Einstein Bots" | Out of scope | AI/Einstein add-on implementation |
| "CTI integration" | Out of scope | Requires telephony partner and custom dev |
| "Custom objects" | Limited | 1-2 if critical, otherwise custom project |
| "API integrations" | Limited | Basic REST/SOAP only, complex = custom project |
| "Portal/Community" | Limited | Basic Experience Cloud setup, custom = separate project |
| "Unlimited processes" | Limited | Up to 10 automations, more = maintenance issues |

### Implementation Notes

- Transform tier is for operational complexity, not custom development
- Skills-based routing requires clear skill taxonomy and agent profiling
- Case swarming works best for complex technical issues, not all cases
- Service Contracts require product catalog and pricing data
- Multiple service processes need clear differentiation criteria
- Approval workflows should focus on high-value scenarios (discounts, SLA exceptions)
- Advanced automation limit of 10 processes requires prioritization
- Customer surveys should launch post-implementation, not day 1
- Scheduled reports/dashboards focus on executive and management views
