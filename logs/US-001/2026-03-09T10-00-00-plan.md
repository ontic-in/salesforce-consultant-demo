### Phase 2: Profile Setup

**Goal:** Ensure `Service_Cloud_Agent` custom profile exists with correct restrictions: no ModifyAllData, no ViewAllData, no Case delete.

> **Note:** A `Service_Cloud_Agent` profile already exists in the org (ID: `00eHo000000kDCeIAM`) from a prior engagement. We will retrieve it, verify its permissions, and modify if needed.

**Step 2.1 — Retrieve existing profile**

```bash
sf project retrieve start --metadata "Profile:Service_Cloud_Agent" --target-org exo-sf-org
```

**Step 2.2 — Audit profile permissions**

After retrieval, check the profile XML for:

```bash
# Check for ModifyAllData (should NOT be present or should be false)
grep -A1 "ModifyAllData" force-app/main/default/profiles/Service_Cloud_Agent.profile-meta.xml

# Check for ViewAllData (should NOT be present or should be false)
grep -A1 "ViewAllData" force-app/main/default/profiles/Service_Cloud_Agent.profile-meta.xml

# Check Case delete permission (should be false)
grep -B5 -A5 "Case" force-app/main/default/profiles/Service_Cloud_Agent.profile-meta.xml | grep -A1 "Delete"
```

**Step 2.3 — Modify profile if needed**

If any of the following are true, modify the profile XML:
- `<name>ModifyAllData</name>` with `<enabled>true</enabled>` → set to `false`
- `<name>ViewAllData</name>` with `<enabled>true</enabled>` → set to `false`
- Remove all dependent permissions if disabling ModifyAllData: `ManageDataIntegrations`, `CreatePackaging`, `InstallPackaging`, `DataExport`
- Case `<objectPermissions>` has `<allowDelete>true</allowDelete>` → set to `false`

> **Gotcha (from guide):** Permission Sets cannot revoke access. Delete prevention MUST be enforced at the profile level. The profile's `<allowDelete>false</allowDelete>` on Case objectPermissions is the only way to prevent deletion.

**Step 2.4 — Deploy updated profile**

```bash
sf project deploy start --source-dir force-app/main/default/profiles/Service_Cloud_Agent.profile-meta.xml --target-org exo-sf-org
```

> **Gotcha (from guide):** Use `--source-dir` (not `--manifest`) when deploying from an edited file to avoid stale source resolution.

**Step 2.5 — Verify profile**

```bash
sf data query --query "SELECT Id, Name FROM Profile WHERE Name = 'Service Cloud Agent'" --target-org exo-sf-org
```

```bash
sf data query --query "SELECT SobjectType, PermissionsDelete FROM ObjectPermissions WHERE Parent.Profile.Name = 'Service Cloud Agent' AND SobjectType = 'Case'" --target-org exo-sf-org
```

**Expected:** Profile exists, Case `PermissionsDelete = false`.

### Phase 3: OWD Changes

**Goal:** Set Case OWD to Private, Account OWD to Public Read Only.

> **Important:** Deploy Account and Case OWD changes together in a single deployment. Deploying separately risks intermediate state errors and concurrent sharing recalculation conflicts.

**Step 3.1 — Retrieve Account and Case object metadata**

```bash
sf project retrieve start --metadata "CustomObject:Account" --metadata "CustomObject:Case" --target-org exo-sf-org
```

**Step 3.2 — Verify current sharingModel values**

```bash
grep "sharingModel" force-app/main/default/objects/Account/Account.object-meta.xml
grep "sharingModel" force-app/main/default/objects/Case/Case.object-meta.xml
```

**Expected:** Both currently show `<sharingModel>ReadWrite</sharingModel>` (matching SOQL result `Edit`).

> **Gotcha (from guide):** SOQL returns `Edit`/`Read`/`None` but XML uses `ReadWrite`/`Read`/`Private`. Don't confuse the two.

**Step 3.3 — Modify sharingModel values**

In `Account.object-meta.xml`:
```xml
<sharingModel>Read</sharingModel>
```

In `Case.object-meta.xml`:
```xml
<sharingModel>Private</sharingModel>
```

Also update `<externalSharingModel>` to match if present.

**Step 3.4 — Deploy OWD changes**

```bash
sf project deploy start --source-dir force-app/main/default/objects/Account/Account.object-meta.xml --source-dir force-app/main/default/objects/Case/Case.object-meta.xml --target-org exo-sf-org
```

**Step 3.5 — Verify OWD (SOQL)**

```bash
sf data query --query "SELECT DefaultAccountAccess, DefaultContactAccess, DefaultCaseAccess FROM Organization" --target-org exo-sf-org
```

**Expected:** `DefaultAccountAccess = Read`, `DefaultCaseAccess = None`, `DefaultContactAccess = ControlledByParent`

> **Gotcha (from guide):** SOQL may show old values temporarily due to async sharing recalculation. If values appear unchanged, wait 30 seconds and re-query. Can also verify via metadata retrieve to confirm the deployed value.

**Step 3.6 — Verify via metadata retrieve (backup check)**

```bash
sf project retrieve start --metadata "CustomObject:Case" --target-org exo-sf-org --output-dir /tmp/owd-verify
grep "sharingModel" /tmp/owd-verify/force-app/main/default/objects/Case/Case.object-meta.xml
```

## Known Gotchas (From Feature Guides)

1. **Permission Sets cannot revoke access** — Delete prevention MUST be at profile level, not permission set
2. **SOQL ↔ XML value mismatch** — SOQL returns `Edit`/`Read`/`None`; XML uses `ReadWrite`/`Read`/`Private`
3. **Sharing recalculation lag** — SOQL may show old OWD values for up to 30 seconds after deploy
4. **Profile deploy: use `--source-dir` not `--manifest`** — `--manifest` resolves from `force-app/` and may ignore edits
5. **Create profile before users** — Users should be created AFTER the profile is deployed/verified
6. **ModifyAllData dependency chain** — Disabling ModifyAllData requires also disabling dependent permissions
