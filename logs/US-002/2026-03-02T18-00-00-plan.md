## Requirements Summary

Relay Logic uses a tiered customer model (Silver, Gold, Platinum) that drives SLA response times, case priority, VIP detection, and escalation rules. Two custom fields are needed on the Account object:

1. **Customer_Tier__c** — Picklist (Silver, Gold, Platinum), required. Foundation for downstream automation and routing.
2. **Account_Start_Date__c** — Date, not required. Powers the "new customer within 30 days" auto-prioritization flow (US-004).

**Design Decision (Resolved):** All users get Read/Edit on both fields via a single permission set. Customer Tier is NOT restricted to leads/managers only.

**Design Decision (From Story):** Use custom `Account_Start_Date__c` rather than standard `CreatedDate` — accounts created during migration would all appear "new" otherwise.

### Step 2: Create Custom Field Metadata XML

Create field XML files in the project's `force-app/` directory:

**File:** `force-app/main/default/objects/Account/fields/Customer_Tier__c.field-meta.xml`
```xml
<?xml version="1.0" encoding="UTF-8"?>
<CustomField xmlns="http://soap.sforce.com/2006/04/metadata">
    <fullName>Customer_Tier__c</fullName>
    <label>Customer Tier</label>
    <type>Picklist</type>
    <required>true</required>
    <valueSet>
        <restricted>true</restricted>
        <valueSetDefinition>
            <sorted>false</sorted>
            <value>
                <fullName>Silver</fullName>
                <default>false</default>
                <label>Silver</label>
            </value>
            <value>
                <fullName>Gold</fullName>
                <default>false</default>
                <label>Gold</label>
            </value>
            <value>
                <fullName>Platinum</fullName>
                <default>false</default>
                <label>Platinum</label>
            </value>
        </valueSetDefinition>
    </valueSet>
</CustomField>
```

**File:** `force-app/main/default/objects/Account/fields/Account_Start_Date__c.field-meta.xml`
```xml
<?xml version="1.0" encoding="UTF-8"?>
<CustomField xmlns="http://soap.sforce.com/2006/04/metadata">
    <fullName>Account_Start_Date__c</fullName>
    <label>Account Start Date</label>
    <type>Date</type>
    <required>false</required>
</CustomField>
```

### Step 7: Add Fields to Account Page Layout and Deploy

Edit the retrieved layout XML to add both custom fields. Add them to an appropriate section (typically the Account Information or a new "Customer Details" section).

Add these `<layoutItems>` entries within a `<layoutColumns>` section:

```xml
<layoutItems>
    <behavior>Required</behavior>
    <field>Customer_Tier__c</field>
</layoutItems>
<layoutItems>
    <behavior>Edit</behavior>
    <field>Account_Start_Date__c</field>
</layoutItems>
```

> **Note:** `Customer_Tier__c` uses `<behavior>Required</behavior>` to enforce the field-level required constraint on the layout. `Account_Start_Date__c` uses `<behavior>Edit</behavior>` since it is not required.

Deploy the updated layout:

```bash
sf project deploy start --source-dir "force-app/main/default/layouts/Account-Account Layout.layout-meta.xml" --target-org deborah@realfast.ai.deb-testing
```

## Known Gotchas

| # | Gotcha | Source | Mitigation |
|---|--------|--------|------------|
| 1 | FLS defaults to zero access — new custom fields are invisible until Permission Set grants access | Field Management guide | Always deploy Permission Set alongside fields (Steps 3-4 sequenced) |
| 2 | SOQL verification may show stale values for ~30 seconds after deploy | Learning from US-001 | Wait 30 seconds before re-querying; check `sf project deploy report` to confirm deploy first |
| 3 | Required picklist fields may fail deploy if FLS is included in field XML | Field Management guide | Use page layout `Required` behavior instead of field-level `required` + separate FLS in Permission Set |
| 4 | FieldPermissions uses standard API only (no `--use-tooling-api` flag) | Permission Management guide | Omit `--use-tooling-api` from FieldPermissions queries |
| 5 | Layout field not visible even after deploy | Page Layouts guide | Both FLS AND layout placement are required for visibility |
| 6 | `restricted=true` on picklist prevents adding values via UI | Salesforce standard behavior | Intended — protects data integrity for tier-driven automation |

## Design Decisions Log

| Decision | Choice | Rationale |
|----------|--------|-----------|
| FLS access model | Open to all (single permission set) | Consultant decision — all 7 users get Read/Edit on both fields. Simpler permission model. |
| Customer_Tier__c required enforcement | Field-level `required=true` + layout `Required` behavior | Belt-and-suspenders: field metadata AND layout both enforce requirement |
| Picklist restricted | `restricted=true` | Protects data integrity — downstream automations (SLA, VIP detection) depend on exact values |
| Account_Start_Date__c vs CreatedDate | Custom field | Migration scenario — CreatedDate would make all migrated accounts appear "new" |
| No custom Contact fields | Skipped | AI suggestion only, no confirmed business requirement |
