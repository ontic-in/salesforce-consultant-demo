## Requirements Summary

Implement three record-triggered flows forming Relay Logic's core automation layer:

1. **VIP Multi-Case Detection** (Flow 1): When a Gold/Platinum customer creates a case and already has open cases (2+ total), set Priority=Critical, route to Senior Queue, post Chatter alert.
2. **New Customer Priority Bump** (Flow 2): When a case is created for a new customer (account < 30 days old) and priority isn't already Critical, set Priority=High.
3. **Auto-Create Follow-Up Task** (Flow 3): When a case is assigned to a user (not queue), create "Initial Response Required" task with priority-based due date.

**Execution order**: Flow 2 (Before Save) runs first, then Flow 1 (After Save) can override to Critical. This ensures VIP detection takes precedence.

**Scope addition**: This story also includes creating/verifying all four queues (Senior, Support, Engineering, Shipping) with members drawn from existing org users.

### Step 4: Deploy Flow 1 — VIP Multi-Case Detection

**Purpose:** After-Save record-triggered flow on Case insert. Detects VIP customers (Gold/Platinum) with 2+ open cases, sets Priority=Critical, routes to Senior Queue, posts Chatter alert.

**Flow Logic:**
```
Trigger: Case — After Insert
Entry Condition: {!$Record.Account.Customer_Tier__c} IN ('Gold', 'Platinum')

1. Get Records: Open cases on same Account
   Object: Case
   Filter: AccountId = {!$Record.AccountId}
           AND Status NOT IN ('Resolved', 'Closed')
           AND Id != {!$Record.Id}
   Store: varOpenCases (collection), varOpenCaseCount (count)

2. Decision: Has Multiple Open Cases?
   Condition: varOpenCaseCount >= 1 (1 existing + 1 new = 2+ total)

   YES path:
   3a. Update Records: Set current case
       Priority = 'Critical'
       OwnerId = <SENIOR_QUEUE_ID>

   3b. Create Records: FeedItem (Chatter post)
       ParentId = {!$Record.Id}
       Body = 'VIP Alert: {tier} customer {Account.Name} with {count+1} open cases — routed to senior queue.'

   NO path: (end — no action needed)

Fault Path: Log error, do not block case creation
```

**4a. Build flow XML:**

Create file `force-app/main/default/flows/VIP_Multi_Case_Detection.flow-meta.xml` with:
- `<processType>AutoLaunchedFlow</processType>`
- `<triggerType>RecordAfterSave</triggerType>`
- `<start>` with `<object>Case</object>`, `<recordTriggerType>Create</recordTriggerType>`
- Entry condition: `Account.Customer_Tier__c` IN (Gold, Platinum)
- Decision, Get Records, Update Records, Create Records (FeedItem) elements per logic above
- `<status>Active</status>` — **CRITICAL: must be Active, not Draft**

### Step 5: Deploy Flow 2 — New Customer Priority Bump

**Purpose:** Before-Save record-triggered flow on Case insert. Sets Priority=High for new customers (account < 30 days) when priority is not already Critical.

**Flow Logic:**
```
Trigger: Case — Before Insert
Entry Condition: {!$Record.Priority} != 'Critical'

1. Get Records: Parent Account
   Object: Account
   Filter: Id = {!$Record.AccountId}
   Store: varAccount

2. Formula: Is New Customer?
   {!varAccount.CreatedDate} >= {!$Flow.CurrentDate} - 30

3. Decision: Is Account New?
   Condition: Formula result = true

   YES path:
   4. Assignment: Set Priority = 'High' on $Record
      (Before-Save — no DML needed, modifies in-memory)

   NO path: (end — keep default priority)
```

### Step 6: Deploy Flow 3 — Auto-Create Follow-Up Task

**Purpose:** After-Save record-triggered flow on Case insert and update. When OwnerId changes to a User (not Queue), creates "Initial Response Required" task with priority-based due date.

**Flow Logic:**
```
Trigger: Case — After Insert or Update
Entry Condition:
  - OwnerId isChanged (for updates; always true for inserts)
  - OwnerId starts with '005' (User ID prefix, not '00G' Queue prefix)

1. Formula: Calculate Due Date
   IF({!$Record.Priority} = 'Critical', TODAY(),
   IF({!$Record.Priority} = 'High', TODAY() + 1,
   TODAY() + 3))

2. Create Records: Task
   Subject = 'Initial Response Required'
   WhoId = {!$Record.ContactId}
   WhatId = {!$Record.Id}
   OwnerId = {!$Record.OwnerId}
   Priority = {!$Record.Priority}
   ActivityDate = {!formulaDueDate}
   Status = 'Not Started'
```

- Entry condition: OwnerId isChanged AND OwnerId STARTS_WITH '005'

## Known Gotchas

| # | Gotcha | Source | Mitigation |
|---|--------|--------|------------|
| 1 | Flow XML must have `<status>Active</status>` — deploying as Draft means flow exists but doesn't fire | automations-flows guide | Always set Active in XML; verify with FlowDefinitionView post-deploy |
| 2 | StandardValueSet deploys are full-replace — always retrieve before modifying | rollback guide | Retrieve CasePriority first if modification needed |
| 3 | CLI may report `MetadataTransferError` even on successful deploys | automations-flows guide | Verify with FlowDefinitionView query, don't trust CLI exit code |
| 4 | `--source-dir` deploys ALL flows in directory — pre-existing broken flows cause failure | automations-flows guide | Use `--metadata "Flow:<ApiName>"` for single-flow deploys |
| 5 | Flow deactivation via metadata (Draft/Obsolete status) is unreliable | automations-flows guide, rollback guide | Use Tooling API `FlowDefinition` update with `activeVersionNumber:0` |
| 6 | Before-Save flows cannot do DML — only modify triggering record in-memory | automations-flows guide | Flow 2 uses assignment elements only (no Create/Update Records) |
| 7 | After-Save flows that update the triggering record can cause recursion | Salesforce best practice | Flow 1 updates Case (Priority + OwnerId) — entry conditions prevent loop |
| 8 | OwnerId prefix '005' = User, '00G' = Queue — entry condition for Flow 3 | Solution design | Use STARTS_WITH operator on OwnerId to distinguish user vs queue |
| 9 | Chatter (FeedItem) creation requires Chatter to be enabled in org | General Salesforce | Verify Chatter is enabled before deploying Flow 1 |
| 10 | Duplicate GroupMember creation throws error | General Salesforce | Check existing members before adding; skip if already present |

## Rollback Plan

If any flow causes issues after deployment:

**Deactivate a specific flow:**
```bash
# Step 1: Get FlowDefinition Id
sf data query --query "SELECT Id, DeveloperName FROM FlowDefinition WHERE DeveloperName = '<FLOW_API_NAME>'" --use-tooling-api --target-org paulomi

# Step 2: Deactivate
sf data update record --sobject FlowDefinition --record-id <ID> --values "Metadata='{\"activeVersionNumber\":0}'" --use-tooling-api --target-org paulomi

# Step 3: Verify
sf data query --query "SELECT ApiName, ActiveVersionId FROM FlowDefinitionView WHERE ApiName = '<FLOW_API_NAME>'" --target-org paulomi
```

**Remove a queue member:**
```bash
sf data query --query "SELECT Id FROM GroupMember WHERE GroupId = '<QUEUE_ID>' AND UserOrGroupId = '<USER_ID>'" --target-org paulomi
sf data delete record --sobject GroupMember --record-id <MEMBER_ID> --target-org paulomi
```
